# Mycel 要補上的空缺

狀態：說明文件

這份文件說明：為什麼過去較少出現像 Mycel 這樣的協定，以及 Mycel 想處理的問題在今天通常是如何被處理。

## 1. 為什麼以前少有像 Mycel 這樣的協定

Mycel 想處理的是一個很尷尬的交界問題：

- 比一般文件協作更嚴格
- 比區塊鏈更不想要全網單一共識
- 比 Git 更在意讀者端的預設採用結果
- 比一般 app backend 更在意可驗證歷史與可攜性

這類需求一直存在，但通常不夠集中成一個獨立的協定市場，所以過去多半被拆散處理。

### 1.1 問題是分散的，不是單一產業的集中痛點

多數使用者一開始真正需要的是：

- 好用的編輯器
- 文件版本控制
- 中央化後台
- 網頁協作

很少人會一開始就同時要求：

- 可驗證歷史
- 多分支合法並存
- 預設閱讀版本要由固定規則導出
- 還要支援去中心化複製

### 1.2 現成方案各自吃掉一部分需求

因此，市場通常不會自然逼出像 Mycel 這樣的中間層協定。

### 1.3 這類協定不容易講成一句話

Git 很好懂，區塊鏈也很好講。

但 Mycel 這種「不是 Git、不是鏈，而是可驗證文字歷史與閱讀治理協定」比較不容易一眼理解。

### 1.4 工程成本不小，但短期商業回報不明顯

要做出 Mycel 這種系統，需要同時處理：

- canonicalization
- signed objects
- replay
- accepted-state rules
- partial replication
- wire sync

這很重，但不像聊天 app、支付系統或內容平台那麼容易快速變現。

## 2. 今天這些問題通常怎麼被處理

### 2.1 中央化平台

常見做法是使用：

- Google Docs
- Notion
- Confluence
- 各種 CMS 與知識庫平台

這類系統通常：

- 由平台資料庫保存所有狀態
- 由平台決定預設版本
- 可以看歷史，但通常不具備獨立可驗證、可攜帶的歷史模型

好處：

- 好用
- 部署快
- 使用者學習成本低

限制：

- 平台依賴強
- 歷史難以獨立驗證
- 分支與治理模型通常較弱

### 2.2 Git

另一種常見做法是把文本當成 Git repository 來處理。

這樣可以得到：

- branch
- commit history
- merge
- review flow

好處：

- 很適合作者與維護者
- 歷史保留能力強
- 多分支模型成熟

限制：

- 比較像作者工具，不一定適合一般讀者
- 「預設閱讀版本如何導出」不是它的核心設計
- app / governance / accepted-state 的語意不容易自然表達

### 2.3 區塊鏈或 smart contract

有些系統會把重要狀態寫上鏈，再用鏈上共識決定唯一狀態。

好處：

- 強一致性
- 高可驗證性
- 一致的全域狀態

限制：

- 太重
- 成本高
- 對很多文字治理情境是過度設計
- 很多情況根本不需要全網只有一個唯一真版本

### 2.4 客製後端與資料庫

現實世界最常見的做法，其實是自己寫一個 app backend：

- 狀態存在資料庫
- 審計靠 audit log
- 權限、閱讀結果、歷史都放在應用後端

好處：

- 最實際
- 最好整合既有產品
- 最快上線

限制：

- 很少可互通
- 可攜性差
- protocol-level verification 幾乎不存在

## 3. Mycel 想補的是哪個空洞

Mycel 想補上的，是中央平台與全域共識鏈之間的那一層：

- 歷史要可驗證
- 預設採用結果要可解釋
- 多個有效分支要能並存
- 又不想付出區塊鏈那種全域共識成本

換句話說，Mycel 想提供的是：

- 可驗證歷史
- 依固定規則導出的預設閱讀版本
- 去中心化複製
- 但不要求全網只有一個唯一真版本

它不是要取代所有現有方案，而是要把這些零散需求，變成一套有分層、可規範、可互通的協定設計。

## 4. 一句話版本

過去不是沒有這種需求，而是它一直被中央平台、Git、區塊鏈與客製後端各自用不完整的方式吸收掉了。Mycel 的價值，在於把這些被拆散的需求重新組成一套可驗證、可導出預設閱讀、又能去中心化複製的協定。
