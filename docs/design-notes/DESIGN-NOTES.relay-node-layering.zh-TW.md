# Relay Node 分層位置

狀態：設計備忘錄

這份文件定義 `relay node` 在 Mycel 分層模型中的位置。

這份說明刻意保持範圍狹窄。它不打算一次標準化完整的 relay protocol、relay 探索機制，或 relay 激勵模型。

## 1. 工作定義

`relay node` 是一種偏向傳輸可用性與物件 / session 轉送的節點角色。

它可以協助處理：

- 穩定可達性
- 跨區較低延遲的傳輸
- NAT / 防火牆橋接
- 有界的物件轉送
- 可選的短期傳輸端快取

不應預設它用來定義社會真相、accepted-head selection，或應用領域語意。

## 2. 主要分層位置

`relay node` 這個概念主要屬於 `wire / 傳輸層`。

原因：

- 這個角色處理的是連線行為與資料轉送
- 它影響的是 session 互通，而不是協定核心裡已簽署物件的意義
- 它應該能和治理與應用語意分開

因此，如果 Mycel 之後要正式定義 relay capability，第一個規範落點應該是傳輸協定表面。

## 3. 次要分層位置

### 3.1 Profile 層

部署情境專屬的 relay 政策應放在 profile。

例如：

- 是否允許或要求使用 relay
- 是否要求只能使用 Tor relay
- 哪些 relay 類型可被用來 bootstrap 或 forwarding
- relay 的優先順序、fallback 或在地性政策

這些都不是通用的協定真相；除非跨實作互通真的需要，否則應維持在 profile 層。

### 3.2 本地工具層

relay 的營運支援應放在本地工具層。

例如：

- relay 健康檢查
- relay 延遲排序
- relay 設定輔助
- relay 啟動清單
- 本地 relay 可觀測性

這些都是營運便利功能，不應強制寫進協定物件。

## 4. 不應放進協定核心的內容

`relay node` 這個概念不應主要放在 `協定核心`。

協定核心不應把 relay node 定義成：

- 治理權威
- accepted-state selector
- 來源真相來源
- 應用領域角色

協定核心裡的已簽署物件，必要時可以參照傳輸資訊；但 relay 的角色語意不應預設變成已簽署的社會真相。

## 5. Relay Node 為何能加速

relay node 有用，不需要依賴更強的硬體設備。

它能提升效能，主要是因為：

- 部署在更好的網路位置
- 長時間穩定在線
- 專心處理轉送，而不是完整本地工作流程
- 在跨區之間有更好的連線
- 能擔任穩定的中途傳輸點

它的主要優勢來自網路拓樸與可用性，而不是原始算力。

## 6. 目前建議的 Mycel 立場

目前建議：

- 先把 `relay node` 維持在設計備忘錄層
- 把它未來正式規範的主要落點視為傳輸協定工作
- 將部署情境專屬的 relay 規則放在 profile
- 將 relay 的營運行為放在本地工具

這樣能維持範圍紀律，等 Mycel 進到 `M4` 的 wire 與 peer interoperability 階段，再決定是否需要更正式的傳輸角色模型。
