Once a team starts using multiple coding agents in one repository, it quickly
learns that "the code changed" is not enough information.

Teams also need to know:

- which agent spent the effort
- how much work a cycle consumed
- which model and reasoning mode produced the result
- where one owned work item ended and the next one began

That is why token usage and commit trailers belong inside the workflow layer.

They are not decorative metadata.
They are part of how the repo stays understandable to both humans and agents.

## The Coordination Problem Behind The Metadata

In a single uninterrupted chat, it is easy to treat token usage and commit
trailers as optional detail.

In a real multi-agent repo, they do more than annotate history.
They answer coordination questions:

- was this work item small or unexpectedly expensive
- did the agent really close the cycle it claimed to close
- which model and effort level produced this landed change
- can another agent continue safely without guessing the prior boundary

That is why we treat usage and trailer fields as workflow surfaces, not just
logging output.

## The Tools We Use

The main repo-native tools in this part of the workflow are:

- `scripts/agent_work_cycle.py`
- `scripts/codex_token_usage_summary.py`
- `scripts/agent_safe_commit.py`
- `scripts/codex_thread_metadata.py`

They do different jobs, but they fit together.

`agent_work_cycle.py` records begin and end boundaries for a user-command
cycle. During closeout it can surface a usage summary like `+53K this cycle
est.` so the cycle leaves behind both lifecycle state and cost context.

`codex_token_usage_summary.py` is the helper that turns session evidence into
per-turn or per-cycle token summaries instead of leaving usage trapped inside
raw session files.

`codex_thread_metadata.py` exposes the current model and reasoning metadata in
a way other scripts can reuse.

`agent_safe_commit.py` uses that metadata when it creates an owned commit and
adds commit trailers such as:

- `Agent-Id`
- `Model`
- `Reasoning-Effort`
- `Token-Spent` when usage evidence is available

The result is one workflow surface spread across timestamps, checklists, and
Git history instead of one hidden runtime state nobody can inspect later.

## Why Token Usage Belongs In The Workflow

Token usage matters because workflow is not only about correctness.
It is also about operational cost and scope.

If one cycle quietly burns far more tokens than expected, that can signal:

- an unclear task boundary
- repeated failed attempts
- an overly large context load
- a workflow that should probably have been split earlier

That information helps humans review how work happened.
It also helps later agents inherit better expectations about slice size and
handoff discipline.

In other words, usage is not just finance reporting.
It is coordination feedback.

## Why Commit Trailers Matter

Commit trailers do similar work for landed history.

Without them, a commit may show what changed but still hide important context:

- which agent owned the slice
- which model produced it
- how much token spend the cycle likely consumed
- whether the reasoning setting was lightweight or deeper

That missing context forces later readers to reconstruct intent from chat
memory, issue threads, or guesswork.

With trailers, the commit boundary becomes more than a diff.
It becomes a compact continuation record.

That is especially useful when a later agent needs to:

- audit a previous work item
- compare outcomes across models or effort settings
- understand whether a costly cycle justified its result
- continue follow-up work without losing authorship boundaries

## Human-Readable And Agent-Usable At The Same Time

This is the same design principle we use for checklists and handoffs.

The metadata should stay useful to both audiences:

- humans can read timestamp lines, commit trailers, and summaries directly
- agents can parse the same fields to preserve identity and continuation context

That dual-use property matters.

If the data is only machine-friendly, humans stop trusting it because they
cannot inspect it easily.
If it is only prose for humans, agents cannot reuse it safely in automation.

The workflow is strongest when the same artifact works for both.

## The Tradeoff

None of this is free.

Adding token summaries and commit trailers means more explicit process.
It makes the workflow more opinionated.
It asks the repo to preserve metadata that some lightweight agent demos would
skip entirely.

But the upside is clarity:

- cost becomes visible
- ownership becomes durable
- model context survives beyond one chat
- landed history becomes easier to audit

That is usually a good trade once multiple agents and multiple sessions are
normal.

## Takeaway

Token usage and commit trailers are not side channels.

They are part of the coordination layer:

- workcycle timestamps make usage visible
- helper scripts summarize token spend
- safe commits preserve agent and model identity
- commit trailers turn landed history into a more usable workflow artifact

That is the broader Mycel pattern:
important workflow context should not disappear just because a chat ends.
