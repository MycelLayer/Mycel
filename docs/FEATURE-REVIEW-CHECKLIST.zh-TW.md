# Mycel 功能審查檢查清單

狀態：目前有效的工作檢查清單

這份文件是用來反覆審查任何新 Mycel 功能提案的檢查清單。

當我們要判斷一個功能應該放在哪一層時，請使用這份清單：

- 協定核心
- 傳輸協定
- profile 層
- 應用層
- 僅限本地工具

它的目的是讓功能成長維持保守、可稽核，並兼顧隱私。

## 1. 快速關卡

在投入太多實作成本前，先問：

1. 這個功能是否明確解決了目前真實存在的問題？
2. 它是否屬於目前的 milestone？
3. 它現在是否可以先不放進協定核心？
4. 它是否保留決定性的驗證結果？
5. 它是否會削弱隱私或匿名性？

如果答案不明確，先放慢速度，把功能範圍縮小。

## 2. 協定核心審查概念

每次都要檢查下面這些概念。

### 2.1 匿名性與隱私

- 這個功能是否會更常暴露穩定的節點身分？
- 它是否讓作者、讀者、維護者或 relay 更容易被關聯？
- 它是否把單跳傳輸中繼資料變成長期保存的來源脈絡？
- 它是否加入本來應維持可選的來源追蹤？

預設偏向：

- 讓協定核心維持中立
- 避免在協定物件中留下永久的來源追蹤
- 優先使用有界、本地或需明確選用的稽核表面

### 2.2 分層位置

先問這個功能真正屬於哪一層：

- `協定核心`：跨實作互通的簽署物件規則
- `傳輸協定`：傳輸 / session 的互通規則
- `profile 層`：部署情境專屬的政策
- `應用層`：領域語意
- `本地工具`：僅供操作者方便使用

預設偏向：

- 如果它不是跨實作互通所必需，就不要放進協定核心

### 2.3 已簽署真相與衍生狀態

- 哪些欄位必須簽署？
- 哪些值必須可重新計算？
- 哪些摘要只是快取或索引？
- 未簽署的輔助資料是否可能意外凌駕 canonical signed objects？

預設偏向：

- 先以已簽署的來源物件為準
- 衍生 view、索引與報告放在第二順位

### 2.4 決定性

- 不同節點是否仍會算出相同的 ID、簽章、replay 結果與選定輸出？
- canonicalization 是否仍然穩定？
- 這個功能是否依賴本地時鐘、排序偶然性或環境差異？

預設偏向：

- 相同輸入應在所有地方得到相同驗證結果

### 2.5 治理邊界

- 這個功能是否影響 accepted-head selection？
- 它是否引入新的治理訊號？
- 它是否讓非治理資料影響 view maintainer authority？
- 它是否模糊了編輯活動與 selector weight 的邊界？

預設偏向：

- 讓治理輸入保持狹窄且明確

### 2.6 部分複製

- 在沒有全域完整資料集的情況下，節點是否仍能運作？
- 這個功能是否迫使所有節點都儲存全部物件？
- 缺失資料是否能清楚失敗，而不是默默造成行為損壞？

預設偏向：

- 保留有界同步與部分複製

### 2.7 可稽核性與可追蹤性

- 我們能否稽核正確性，而不必永久識別誰最先 relay 或託管某個內容？
- 這個功能增加的是可驗證歷史，還是只是更多可追蹤性？
- 相同需求是否能用本地日誌或需明確選用的 profile 來滿足？

預設偏向：

- 優先保留可稽核狀態，而不是全域可追蹤的來源軌跡

### 2.8 失敗模型

- 資料缺失時會發生什麼？
- 資料格式錯誤時會發生什麼？
- 依賴物件未簽署、不可驗證或不可解析時會發生什麼？
- peer 或節點說謊時會發生什麼？

預設偏向：

- 儘早明確規定負面情境
- 適合時補上 fixture-backed 且 CLI 可見的失敗案例

### 2.9 面向使用者的表面一致性

- parser、verify、inspect、store 與 reader 表面是否維持一致？
- 若驗證變得更嚴格，面向使用者的輸出是否會清楚說明？
- 新的失敗情況是否有在 CLI 層級的測試中暴露？

預設偏向：

- 不要只修內部層，而把面向使用者的表面留在後面

### 2.10 範圍紀律

- 這個功能是否足夠精簡？
- 它是否偷偷夾帶了第二個問題？
- 我們能否先交付較窄的版本？
- non-goals 是否已經寫清楚？

預設偏向：

- 先做窄版，再逐步擴大

## 3. 反覆審查問題

當我們重新審視某個功能時，再問一次這六個問題：

1. 它是否削弱匿名性或隱私？
2. 它是否放在正確的層？
3. 哪些是已簽署真相，哪些只是衍生結果？
4. 所有節點是否仍會得到相同的決定性結果？
5. 它是否意外改變治理行為？
6. 在缺失、格式錯誤或惡意輸入下，它的失敗模式是什麼？

## 4. 決策經驗法則

除非有很強的理由，否則預設採用以下經驗法則：

- 把領域專屬語意放進 profile 或應用層。
- 讓協定核心的物件規則保持小而穩定。
- 讓傳輸協定專注於 session 互通，而不是社會真相。
- 當隱私重要時，優先採用本地日誌，而不是協定層的全域來源脈絡。
- 優先補明確的負面測試，而不是依賴非正式假設。
- 優先做一批範圍窄的功能，而不是一包過大的功能集合。

## 5. 最低功能提案內容

在開始實作前，一個新功能提案通常至少應回答：

- 問題：

- 已簽署欄位：
- 衍生欄位：
- 隱私影響：
- 治理影響：
- 失敗模式：
- 驗證計畫：
